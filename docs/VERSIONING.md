# Versioning

## Важное правило

Все агенты и все следующие сессии должны считать версионирование обязательной частью изменения приложения.

Если меняется пользовательское поведение, оформление, настройки, данные, генерация или сборочная конфигурация, нужно:

1. проверить текущую версию в `DreamAssembler/DreamAssembler.csproj`;
2. при необходимости изменить ее там же;
3. обновить `docs/WORKLOG.md`, если шаг значимый;
4. не придумывать отдельные локальные версии в коде, окнах или markdown-файлах.

Источник правды для версии приложения:

- `DreamAssembler/DreamAssembler.csproj`

Источник правды для версии набора данных:

- `DreamAssembler/Data/data-manifest.json`

## Формат версии

Используется формат из четырех чисел:

`MAJOR.MINOR.PATCH.BUILD`

Пример:

`0.1.0.0`

## Смысл чисел

### `MAJOR`

Менять при крупной смене направления, несовместимом пересмотре структуры приложения или большом пользовательском переходе.

Для MVP обычно меняется редко.

### `MINOR`

Менять при заметных новых возможностях:

- новый режим генерации;
- новая крупная подсистема;
- тема оформления;
- редактор данных;
- новые важные настройки.

### `PATCH`

Менять при рабочих улучшениях без смены фазы продукта:

- улучшение UI;
- заметное улучшение генерации;
- новая группа шаблонов или словарей;
- новые проверки данных;
- изменение архитектуры без резкого пользовательского скачка.

### `BUILD`

Менять чаще всего.

Это число предназначено для мелких правок и косметических сборок:

- маленькие багфиксы;
- pixel-perfect правки;
- выравнивание;
- мелкие улучшения текста;
- аккуратные правки popup, карточек, отступов, цветов;
- небольшие служебные изменения без новой функциональности.

## Как менять версию

### Для маленькой локальной правки

Было:

`0.1.0.0`

Стало:

`0.1.0.1`

### Для заметного улучшения внутри текущей линии

Было:

`0.1.0.4`

Стало:

`0.1.1.0`

При увеличении `PATCH` число `BUILD` сбрасывается в `0`.

### Для новой крупной возможности

Было:

`0.1.3.7`

Стало:

`0.2.0.0`

При увеличении `MINOR` числа `PATCH` и `BUILD` сбрасываются в `0`.

## Версия набора данных

Версия `data-manifest.json` живет отдельно от версии приложения.

Текущая версия набора данных:

`0.6.7`

Используется обычный трехчастный формат:

`MAJOR.MINOR.PATCH`

Пример:

`0.5.5`

### Как понимать числа

#### `MAJOR`

Менять только при несовместимом пересмотре структуры набора данных:

- сломаны старые `dictionarySets`;
- меняются базовые правила manifest;
- требуется миграция загрузчика.

#### `MINOR`

Менять при заметном новом этапе корпуса:

- новый крупный слой curated data;
- новая линия словарной структуры;
- большой semantic expansion, который меняет масштаб набора, а не просто добавляет очередной pack.

#### `PATCH`

Менять для обычного рабочего роста корпуса:

- новый atmospheric pack;
- второй pack существующего manifold;
- расширение словарей без смены схемы;
- content-quality pass по JSON-наборам.

### Практическое правило для текущего проекта

- версия приложения и версия данных не обязаны расти вместе;
- для текущей линии expansion почти все шаги должны идти в `PATCH`, а не в `MINOR`;
- если добавили один новый manifold или второй pack существующего поля, это обычно `PATCH`, а не `MINOR`.
