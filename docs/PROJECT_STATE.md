# Project State

## Где сейчас проект

Проект находится на раннем MVP-этапе. Есть рабочая WPF-оболочка и базовый rule-based генератор без морфологии и без пользовательского редактора данных.

## Что уже сделано

- создано решение с двумя проектами:
  - `DreamAssembler.Core`
  - `DreamAssembler.App`
- добавлен тестовый проект `DreamAssembler.Core.Tests`;
- включены nullable reference types;
- добавлены XML-doc комментарии для публичных типов и методов;
- реализованы модели генерации;
- реализованы загрузчики словарей и шаблонов из JSON;
- реализованы fallback-данные;
- реализован генератор с весами и мягким учетом уровня абсурдности;
- реализован WPF UI с выбором режима, уровня абсурдности и количества результатов;
- добавлены команды генерации, копирования и очистки истории;
- добавлено сохранение пользовательских настроек между запусками;
- улучшены статусы UI: отдельно для данных, настроек и действий пользователя;
- исправлено поведение карточек результатов при узкой ширине окна: длинный текст теперь должен корректно переноситься по строкам;
- добавлены стартовые JSON-данные;
- добавлен базовый пакет документации;
- добавлены первые unit-тесты для ядра;
- расширены словари и шаблоны для более разнообразной генерации;
- добавлены анти-повторы по недавним шаблонам и записям;
- убраны несколько небезопасных шаблонных сочетаний, которые часто ломали русскую фразу без морфологии.
- заметно расширены наборы данных по основным категориям;
- добавлена категория `condition` для безопасных коротких состояний и наблюдений.
- словари перестроены в дерево тематических JSON-файлов с manifest-файлом набора данных;
- в модель данных заложен `slot` для безопасных шаблонных позиций.
- шаблоны теперь могут задавать `slotRequirements`, а генератор умеет их соблюдать;
- генератор усиливает совместимость по тегам между уже выбранными элементами одной фразы.
- добавлен консольный инструмент `DreamAssembler.DataTools` для проверки и статистики данных.
- для `ShortText` введены композиционные роли шаблонов, чтобы снижать повторяемость одинаковых каркасных фраз;
- часть `emotion`-данных разведена по более безопасным slot-подтипам для разных типов фраз.
- `twist` разделен на более безопасные slot-подтипы для персонажных и общих конструкций.
- `concept` разделен на отдельные slot-подтипы для сюжетных тем и рефлексивных конструкций.
- `condition` разделен на стартовые состояния, сценические детали и reveal-состояния.
- в `ShortText` усилен базовый порядок narrative-ролей, чтобы поздние интерпретации и meta-конструкции не лезли слишком рано.
- для сложных мест с придаточной частью введен отдельный slot `place_in_clause`.
- для `action` и `object` введен слой смысловой совместимости через `compat:*`-теги.
- начато реальное расширение боевых JSON-словарей в текущей схеме `slot` / `compat` / `composition`.
- добавлены отдельные режимы `Словосочетание` и `Несколько слов`;
- для них скачаны большие внешние CSV-словари существительных, прилагательных, глаголов и прочих слов из одного источника;
- `Словосочетание` собирается как согласованное прилагательное + существительное;
- `Несколько слов` собирается как короткая 3-4-словная конструкция с глаголом прошедшего времени;
- уровень абсурдности в этих режимах отключен и не влияет на выдачу.
- исправлен сбой режима `Несколько слов`: генератор больше не пытается иногда собрать 5 слов вместо допустимых 3-4;
- главное окно теперь открывается по центру экрана, стартует компактнее и лучше подчеркивает сами результаты;
- вместо неудобного слайдера количества результатов теперь используется дискретный выбор 1-10;
- карточки результатов стали компактнее, перестали бессмысленно тянуться на всю ширину для коротких фраз, а основной текст сделан заметно крупнее и удобнее для чтения.
- отдельная кнопка `Скопировать выбранное` убрана: копирование и удаление результата теперь доступны прямо на карточке и через контекстное меню;
- список результатов получил более мягкое выделение без агрессивного системного синего акцента, а вертикальная прокрутка сделана аккуратнее.
- начат переход на собственный window chrome: системная цветная шапка убрана, кнопки окна встроены в интерфейс, а крупный верхний информационный блок удален;
- в проект добавлены встроенные бесплатные шрифты с кириллицей: `Unbounded` для заголовка и `Inter` для интерфейса;
- в кастомной шапке добавлена кнопка настроек, которая уже дает точку входа для следующего шага с темами и редкими параметрами оформления.
- в настройках появился раздел `About` с отображением версии приложения из сборки;
- введен управляемый каркас оформления: тема и шрифт чтения теперь являются пользовательскими настройками, а не случайными значениями в XAML;
- добавлен встроенный шрифт `Literata` для более мягкого чтения результатов, и выбор шрифта чтения уже вынесен в настройки.
- система тем расширена до нескольких светлых и темных палитр с разным характером, а выбор темы сразу отражается в popup настроек;
- текущая версия приложения поднята до `0.2.0.0`, так как оформление и настройки стали реальной пользовательской возможностью.

## Что проверить при следующем шаге

- насколько хорошие реальные результаты дает уже расширенный набор шаблонов;
- корректно ли переносятся длинные результаты по строкам при сужении окна и ресайзе карточек;
- нужно ли дополнительно улучшать шаблоны для русского языка без морфологии;
- насколько часто в `Sentence` и `ShortText` повторяются одинаковые служебные конструкции вроде "Если пересказывать это..." после введения композиционных ролей;
- где еще остаются семантически слабые сочетания между `action`, `object`, `place`, `condition` и `twist`;
- не слишком ли часто `ShortText` собирается из одних и тех же каркасных предложений вроде "С самого начала было ясно одно..." и "Из-за этого всем стало...";
- какие словарные формы дают неестественные наречные и оценочные связки вроде "опасно любопытно" или "дождливо терпеливо";
- где еще появляются неудачные местоименные ссылки и расплывчатые привязки после разделения `twist` на отдельные slot-подтипы;
- насколько новый разрез `concept_story_frame` и `concept_reflection` уменьшает однотипные и расплывчатые смысловые хвосты;
- хватает ли текущего разреза `condition_initial_state` / `condition_scene_detail` / `condition_reveal_state`, или его надо дробить дальше;
- достаточно ли нынешнего порядка narrative-ролей в `ShortText`, или еще встречаются нелогичные скачки по времени;
- насколько заметно улучшилась совместимость `action` и `object` после введения `compat:*`, и какие пары все еще звучат слабо;
- насколько удачно ведет себя сохранение настроек в реальном использовании;
- достаточно ли понятны статусы и сообщения в UI при ошибках и fallback-сценариях;
- какие категории стоит дальше делить на более безопасные словоформы по позициям в шаблоне;
- где еще не хватает плотности словаря для уменьшения повторов;
- насколько заметно первый большой пакет данных снизил повторы в реальной выдаче;
- достаточно ли текущего покрытия тестами для безопасного расширения логики.
- не слишком ли грязными оказываются реальные выборки из внешних CSV-словарей без дополнительной ручной чистки;
- какие стоп-слова и фильтры нужно добавить для словесных режимов, чтобы убрать местоимения, служебные прилагательные и слишком техничные слова;
- стоит ли позже делить словесные режимы на несколько подрежимов вроде `визуально`, `литературно`, `мрачно`, `бытово`.
- не слишком ли абстрактны и книжны многие существительные и прилагательные в словесных режимах для задач рисования и быстрой фантазии;
- достаточно ли аккуратна новая визуальная иерархия окна, или верхний статусный блок все еще оттягивает на себя слишком много внимания;
- нужен ли отдельный более мягкий визуальный режим карточек для `Словосочетание` и `Несколько слов`, где контент должен почти полностью доминировать над служебным chrome.
- не перегружены ли карточки коротких результатов служебными кнопками и заголовками, и не стоит ли позже перейти к более минималистичному hover-режиму действий.
- насколько удачно WPF реально рендерит встроенные шрифты `Unbounded` и `Inter` на разных экранах и DPI.
- насколько удачно WPF реально рендерит встроенный `Literata` на разных экранах и DPI.
- как лучше организовать систему тем: через словари ресурсов, через отдельные theme-файлы или через сервис применения палитры.
- какие следующие бесплатные шрифты стоит добавлять в curated-набор приложения для чтения, не превращая настройки в свалку системных font family.
- нужно ли дальше расширять набор тем до более смелых художественных градиентов, не ломая читаемость и контраст.

## Следующие рекомендуемые шаги

1. Продолжить усиливать совместимость слотов и снижать частоту повторяющихся каркасных конструкций в `Sentence` и `ShortText`.
2. Дальше разводить слишком рискованные словарные формы по более безопасным подтипам, чтобы не собирать неестественные связки наречий и оценок.
3. Дальше наращивать `compat:*`-разметку и при необходимости вводить более тонкие смысловые группы для `action` и `object`.
4. При необходимости дальше усиливать порядок narrative-ролей в `ShortText`, если визуальные проверки еще показывают нелогичные временные скачки.
5. При необходимости расширить тесты на новые правила генерации.
6. Продолжить наращивать наборы данных и шаблонов по жанрам и сценариям крупными пакетами.
7. Расширить правила валидации и статистики по наборам данных.
8. Продолжить вводить более строгие slot-подкатегории там, где они реально улучшают качество.
9. Позже добавить редактор словарей и шаблонов.
10. Отдельно чистить и, возможно, тематически нарезать большой CSV-лексикон словесных режимов.
11. Дальше дорабатывать визуальную иерархию окна вокруг чтения результата, а не вокруг служебных панелей и статусов.
12. Следующим UI-шагом ввести реальное переключение тем и расширить меню настроек вокруг нового кастомного chrome.
13. Следить за версией приложения через `docs/VERSIONING.md` и менять номер в `DreamAssembler/DreamAssembler.csproj` как обязательное правило.

## Технические ограничения текущей версии

- нет морфологии русского языка;
- нет базы данных;
- нет редактора словарей;
- нет плагинной системы;
- `ShortText` пока собирается как 2-5 независимых предложений без связности.

## Что важно помнить новым сессиям

- не добавлять нейросетевую генерацию;
- не тащить базу данных в MVP;
- не усложнять архитектуру без реальной необходимости;
- не хардкодить большие словари в коде;
- документация и комментарии должны оставаться на русском языке.
